# 题库管理系统接口文档（Markdown 版）

> 本文档的接口描述按你给的截图样式组织（请求信息表 → 请求参数表 → 返回 JSON 示例）。

## 0. 通用说明

### 0.1 Base URL

- 统一前缀：`/api`

### 0.2 认证与鉴权（JWT + RBAC）

- 登录成功后服务端返回 `token`（JWT）。后续请求在 Header 中携带：`Authorization: Bearer <token>`。
- 角色：`STUDENT` / `TEACHER` / `ADMIN`。

### 0.3 通用返回结构

除登录接口外，推荐统一返回：

```json
{
  "success": true,
  "msg": "ok",
  "data": {}
}
```

当发生业务错误时：

```json
{
  "success": false,
  "msg": "错误原因",
  "code": "ERROR_CODE"
}
```

### 0.4 分页返回结构

列表接口默认返回：

```json
{
  "success": true,
  "msg": "ok",
  "data": {
    "page": 1,
    "size": 10,
    "total": 0,
    "list": []
  }
}
```

### 0.5 关键枚举值（和数据库/代码保持一致）

- `questionType`：1=single，2=multiple，3=true_false，4=blank，5=short，6=code，7=code_reading
- `difficulty`：1~5
- `questionStatus`：1=draft，2=published，3=archived
- `analysisSource`：1=manual，2=llm_draft，3=llm_final
- `assignmentPublishStatus`：1=draft，2=published，3=closed
- `attemptType`：1=assignment，2=practice
- `attemptStatus`：1=in_progress，2=submitted，3=grading，4=graded
- `answerStatus`：1=draft，2=submitted
- `gradingMode`：1=auto，2=llm，3=manual
- `appealStatus`：1=pending，2=approved，3=rejected，4=resolved

## 0.6 相比开题报告新增的表/概念说明（重点：`qb_llm_call`）

你的开题报告里已经明确：**主观题用 LLM 结构化 JSON 输出（得分/扣分点/理由/置信度）**，并且要**保留模型原始输出与评分依据**用于复核与申诉。
因此我在数据库与接口层补齐了“证据链 + 可追溯性”相关表：

- `qb_llm_call`：保存一次 LLM 调用的**prompt、原始 response、结构化 response_json、耗时、token、状态**，并通过 `biz_type/biz_id` 关联业务（题目解析生成/主观题评分等）。
- `qb_grading_record.llm_call_id`：把某次评分记录与 LLM 调用绑定，教师复核时能直接看到“模型怎么判的”。
- `qb_attempt_question.snapshot_json / qb_paper_question.snapshot_json`：**题目快照**。发布后题目若被修改，历史作答仍可追溯。
- `qb_assignment_target`：支持“只给某些学生发布作业”（有些课程场景需要）。
- `qb_practice_rule / qb_practice_generation`：自适应抽题需要可配置规则与可追溯生成过程（调参、复现实验很有用）。
- `sys_audit_log / sys_login_log`：用于后台审计（谁在什么时候改了什么、登录是否成功等）。
- `qb_question_case`：为编程/代码题预留测试样例（可选加分点）。

这些扩展不会改变你“题库-组卷-作答-判分-统计-申诉/答疑”的主体结构，只是把工程里必要的可追溯性补齐。

## 12. 千问（Qwen）调用说明（后端集成参数）

本项目的 LLM 调用使用 **通义千问 OpenAI 兼容模式**：

- `base_url`：`https://dashscope.aliyuncs.com/compatible-mode/v1`
- Chat Completions：`POST https://dashscope.aliyuncs.com/compatible-mode/v1/chat/completions`
- 鉴权：HTTP Header `Authorization: Bearer $DASHSCOPE_API_KEY`

### 12.1 推荐的请求体（示例）

```json
{
  "model": "qwen-plus",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "..."}
  ],
  "temperature": 0.2,
  "response_format": {
    "type": "json_object"
  }
}
```

### 12.2 结构化输出（用于“主观题评分 JSON”）

- 千问支持 `response_format.type = json_object` 或 `json_schema`，用于约束模型输出 JSON。
- 若使用 `json_object`，提示词中**必须显式要求输出 JSON**（否则会报错/输出不稳定）。

评分 JSON 建议统一为：

```json
{
  "score": 18,
  "deductions": [
    {"point": "遗漏关键点：数组连续内存", "deduct": 5, "reason": "没有说明数组是连续存储"}
  ],
  "reason": "总体理解正确，但解释不完整",
  "confidence": 0.62
}
```

你开题报告里强调的“置信度阈值 + 分歧触发复核 + 申诉证据链”，在接口与表设计上分别对应：

- 阈值触发：`qb_grading_record.needs_review` + `qb_attempt.needs_review`
- 证据链：`qb_llm_call`（prompt/response/json）+ `qb_grading_record.llm_call_id`
